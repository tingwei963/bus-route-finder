### 3. Bug 調查:直達查詢跑出「不該有的」648
- 使用者回報:起站「捷運新店區公所」、迄站「台大醫院」的直達查詢跑出 648,認為不合理。

### 4. Bug 修復:TDX 流量限制導致結果悄悄漏掉
- 使用者回報:「武功國小→師大分部」模糊搜尋只找到 849、綠13、849屈尺社區,懷疑模糊比對邏輯有問題。
- 實際用原始 TDX 資料驗證,光台北市就有近 30 條路線(648、643、松江新生幹線、羅斯福路幹線⋯)直達,但 App 只顯示新北市那 3 條。
- 根本原因:自己測試同一組 TDX API 時親身踩到「API rate limit exceeded」,而 `_fetchCity`/`_fetchInterCity` 原本設計成「失敗就默默回傳空清單」,導致被限流的縣市資料整批消失、畫面上完全沒有任何警示。
- **修復**(`lib/services/tdx_service.dart`):
  - 對每個縣市/公路客運的抓取結果加 5 分鐘記憶體快取,大幅減少重複查詢造成的 API 呼叫量。
  - 抓取失敗時回傳 `null`(而非空清單),跟「真的查到但沒有路線」區分開來,並把失敗的地區名稱收集成 `failedAreas` 回傳給 UI。
  - `main.dart` 在結果上方加橘色警告文字:「⚠️ OO 資料暫時無法取得(可能是 TDX 流量限制),結果可能不完整」。

### 5. 功能調整:轉乘查詢納入直達
- 使用者反應「一次轉乘會排除掉單程」,希望轉乘分頁也能看到直達選項,不用切回「直達」分頁比對。
- 調整 `findTransferRoutes` 呼叫參數:`一次轉乘` 改成查 0~1 次轉乘(含直達),`多次轉乘` 改成查 0~3 次(含直達+轉1次),不再各自獨立區間。
- 結果顯示上,0 次轉乘的項目 subtitle 顯示「直達」而非「0 次轉乘」。

### 6. Bug 修復:轉乘 BFS 的終點站只會留一條路線
- 修完上一項後測試發現:「武功國小→師大分部,一次轉乘」只列出 648 一條直達路線,但「直達」分頁明明有近 30 條。
- 根本原因:BFS 用 `visited` 記錄「這個站名有沒有被到達過」,終點站一旦被第一條路線標記 visited,後面所有其他路線經過同一個終點站都會被 `continue` 跳過,導致 30 條並列的直達/轉乘選項被壓縮成 1 條。
- **修復**:讓「終點站」的比對邏輯繞過 visited 檢查,允許每一條有效路線都各自留下一筆結果;非終點的中繼站仍然維持只探一次,避免組合爆炸拖慢查詢。

## 驗證方式
每個修復都用同一套流程確認:改完 `flutter analyze` 過 → 關閉舊的 `flutter run -d chrome` 背景程序(避免使用者誤用瀏覽器 F5 看到舊編譯結果)→ 重開新的 dev server 讓瀏覽器彈出新視窗測試 → 確認無誤後 `flutter build apk --release` 重新產生 APK,複製到兩個桌面資料夾。
