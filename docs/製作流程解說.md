## 1. 這個專案在做什麼

輸入「起站」跟「迄站」的站名，App 會去查交通部 TDX 運輸資料流通服務，
找出有經過這兩站、且方向正確（起站站序 < 迄站站序）的公車路線，列出來給你看。
查無結果時，會從查過的所有站名裡找最相似的幾個，當作「你是不是要找」的建議。

整體分成三塊：

1. **Flutter App**（`lib/`）— 前端 UI + 查詢邏輯，可以編譯成 Web、Android、iOS。
2. **後端 Proxy**（`api/token.js`）— 一個 Vercel serverless function，專門負責跟 TDX 換 token。
3. **TDX 開放資料 API** — 交通部提供的公車站序資料來源。

---

## 2. 為什麼需要「後端 Proxy」這一塊？

最初的寫法是 Flutter 直接拿 TDX 的 Client ID / Secret 去換 token。
這在「只有自己電腦上跑」時沒事，但只要做成 Web 版部署到網路上，
**瀏覽器端的程式碼（包含編譯後的 JS）使用者都看得到**，金鑰等於公開了。

解法：金鑰不要放進前端，改放在後端環境變數，前端只跟自己的後端要 token。

```
Flutter App  →  GET /api/token (我方 Vercel function)
                     ↓ 用環境變數裡的 Client ID/Secret 換 token
                   TDX 認證伺服器
                     ↓ 回傳 access_token
Flutter App  ←  access_token
Flutter App  →  帶著 access_token 直接呼叫 TDX 查資料 API
```

對應檔案：`api/token.js`
- 從 `process.env.TDX_CLIENT_ID` / `TDX_CLIENT_SECRET` 讀金鑰（這兩個值只設定在 Vercel 專案後台，不存在 repo 裡）
- 用 `client_credentials` grant type 跟 TDX 換 token，原封不動轉給前端

這是這個專案裡**最重要的安全設計**，原則是：**任何會被打包進前端（網頁/App）的程式碼都當作公開的**。

## 3. 核心查詢邏輯：`lib/services/tdx_service.dart`

### 3.1 Token 快取

```dart
static String? _cachedToken;
static DateTime? _tokenExpiry;
```

拿到 token 後存起來，並記錄到期時間（提早 60 秒視為過期，留緩衝）。
下次查詢如果 token 還沒過期就直接重複用，不用每次查詢都跑一次認證流程。

### 3.2 平行查詢多個縣市

TDX 的公車資料是「**按縣市分開**」的 API（`Bus/StopOfRoute/City/{城市代碼}`），
但公車路線常常跨縣市（例如雙北跨界路線），所以：

- 讓使用者用 `FilterChip` 勾選想查的縣市（預設勾雙北）
- 額外有一個「公路客運」開關，固定查 `Bus/StopOfRoute/InterCity`（不分縣市的客運路線，跨縣市必查）
- 用 `Future.wait(futures)` **平行**送出所有勾選縣市的請求，而不是一個一個循序等待 → 大幅縮短查詢時間

這是這個專案前期踩過的坑：第一版「不分縣市全部 22 個都查」雖然最完整，但太慢，
所以改成讓使用者自己縮小範圍，效能跟完整度自己取捨。

### 3.3 比對邏輯

對每一條路線的站序清單（`Stops`）：
1. 邊掃描邊記錄所有出現過的站名（給後面的「你是不是要找」用）
2. 找出起站名稱第一次出現的 `StopSequence`，跟迄站名稱第一次出現的 `StopSequence`
3. 兩者都找到、且**起站序號 < 迄站序號**，才算一條有效路線（確保方向正確，不會把反方向的路線也列進來）

```dart
if (startSeq != null && endSeq != null && startSeq < endSeq) {
  matches.add(RouteMatch(...));
}
```

### 3.4 「你是不是要找」建議

查無結果時，從掃過的所有站名裡，用一個很簡單的相似度評分排序，取前 5 名：

```dart
static int _similarity(String query, String name) {
  if (name == query) return 1000;                         // 完全一樣，最高分
  if (name.contains(query) || query.contains(name)) return 100; // 互為子字串，次高分
  var score = 0;
  for (final char in query.runes) {                        // 否則數共同字元數
    if (name.contains(String.fromCharCode(char))) score++;
  }
  return score;
}
```

這不是什麼高深演算法（沒有用編輯距離之類的），但對「打錯字/打簡稱」這種情境已經夠用，
而且實作成本極低 —— 這是個刻意的取捨：先求好用，不求精確。

## 4. UI 邏輯：`lib/main.dart`

用標準 Flutter `StatefulWidget` 寫法，沒有用額外的狀態管理套件（單頁面，不需要）。

- `_selectedCities` / `_includeInterCity`：使用者勾選的查詢範圍
- `_search()`：驗證輸入 → 呼叫 `TdxService.findRoutes()` → 把結果塞進 `setState`
- `_runDiagnostics()`：「網路診斷」按鈕，依序 ping Google / TDX / Vercel proxy 三個網址，
  把結果丟到 Dialog 顯示 —— 這是真實裝置測試時為了抓「網路根本連不上」的問題暫時加的除錯工具，
  目前決定先保留著，以防之後在別的裝置上又冒出連線問題。

## 5. 部署流程：GitHub → Vercel

1. **GitHub**：`git init` 後建 repo `tingwei963/bus_route_finder`（私有），因為裡面雖然沒有金鑰了，
   但保持私有比較保險。
2. **Vercel**：Import 這個 GitHub repo。
   - 第一次設定要注意：Vercel 預設可能會把你導去付費的「Pro Trial」團隊空間，
     要切到個人的免費 Hobby namespace 再 import，不然 14 天後會要求綁卡。
   - 在 Vercel 專案設定 → Environment Variables 設定 `TDX_CLIENT_ID` / `TDX_CLIENT_SECRET`。
3. **`vercel.json`** 的關鍵設定：

   ```json
   { "buildCommand": "echo '...'", "outputDirectory": "dist" }
   ```

   **Vercel 的 build 環境裡沒有 Flutter**，所以沒辦法叫它自己跑 `flutter build web`。
   解法是「**本機先建好，再把建好的東西 commit 進 repo**」：

   ```
   flutter build web          # 本機編譯出 build/web/
   複製成 dist/                # 把編譯結果放進 dist/ 並 commit
   git push                    # Vercel 偵測到 push，直接拿 dist/ 當靜態網站發布
   ```

   **要記住的坑**：以後改了 UI 或邏輯，**一定要重新 `flutter build web` 並重新複製到 `dist/`、commit、push**，
   否則網站上看到的還是舊版本 —— 目前沒有 CI 自動做這件事。

## 6. Android APK 建置流程與踩過的坑

1. 裝 Android Studio：`winget install Google.AndroidStudio`
2. 第一次設定精靈**不會**自動裝「Android SDK Command-line Tools」，要自己進 SDK Manager 補裝
3. 跑 `flutter doctor --android-licenses` 同意所有授權
   - 這一步要在 **Bash** 跑（不是 PowerShell），因為要連續餵很多個 `y` 答應授權，
     PowerShell 把輸入 pipe 給互動式 Java 程式時不可靠，常常餵不進去
4. `flutter build apk --release`
   - 踩過一次坑：建置中網路斷線，導致下載到一半的 NDK（`ndk/28.2.13676358`）損毀
     （缺 `source.properties`，報 `CXX1101` 錯誤）
   - 解法：刪掉那個 NDK 資料夾，重新跑一次 build，會自動重新下載一份完整的
5. 建好的 APK 在 `build/app/outputs/flutter-apk/app-release.apk`（約 47MB），
   因為沒有 USB 線，改用雲端硬碟分享給測試者

### 6.1 真機測試抓到的 Bug：完全連不上網路

**現象**：release 版 APK 每個 API 呼叫都失敗，錯誤是 `SocketException: Failed host lookup`。

**根因**：Flutter 的 `AndroidManifest.xml` 預設**沒有**寫 `INTERNET` 權限，
而 **debug build 會自動幫你補上**這個權限（所以開發時測試都正常），**release build 不會**。

**解法**：在 `android/app/src/main/AndroidManifest.xml` 手動加：

```xml
<uses-permission android:name="android.permission.INTERNET" />
```

加完後用 `aapt dump permissions` 確認 APK 裡真的有這個權限，再重新測試，
真機查詢（武功國小 → 萬芳醫院）成功回傳正確路線（羅斯福路幹線、671），
代表 Web、Android 兩個平台的真實流程都跑通了。

**這條的學習重點**：debug 模式測得過，不代表 release 模式也沒問題 ——
正式發布前一定要實際裝 release 版的 APK 在真機上測一次，而不是只在模擬器 / debug 模式上測。

## 7. 整體學習重點總結

| 主題 | 學到的東西 |
|---|---|
| 安全 | 任何打包進前端的東西都視為公開；機密一定要放後端，前端只能拿「換好的結果」 |
| 效能 | API 呼叫能平行就平行（`Future.wait`），但「查全部」不一定是好設計，給使用者自己縮小範圍有時更實際 |
| 部署 | 靜態網站平台（Vercel）不代表會幫你跑你的建置工具鏈；不支援的就只能本機建好、把產物 commit 進去 |
| 行動裝置 | debug / release 設定不是只有「有沒有 console log」這種小差異，權限這種東西也會差，必須拿正式版真機測 |
| 除錯 | 與其每次靠猜，不如直接把「連線診斷」做成 App 裡的一個按鈕，遠端測試者自己點一下就能回報症狀 |

## 8. 之後可以延伸的方向（還沒做）

- `dist/` 自動化建置（目前要手動 `flutter build web` + commit，可以考慮 GitHub Actions）
- 站名比對演算法可以換成真正的編輯距離（Levenshtein），「你是不是要找」會更準
- 「網路診斷」按鈕目前是除錯用的暫時功能，正式上架前要決定是隱藏還是保留
